# UX Audit — CalEvents Discovery
**Date:** 2026-04-21  
**Method:** Code-based (App.tsx, searchEngine.ts, component logic, state flows)  
**Persona:** Berkeley student, first visit, wants to find something to do this week  
**Depth:** Standard

---

## Summary

The core experience is solid — search is fast, cards are readable, the detail panel flows naturally. The biggest issues are around **silent state changes that confuse users** (the today→week fallback, the away-game filter, the double-fired click event), a few **accessibility gaps** (no skip link, cards not labelled for screen readers, tiny tap targets), and one **real bug** (View Details fires the event handler twice).

**One thing to fix first:** The filter bar shows "Today" active while displaying this week's events. This is the most disorienting moment in the whole app — users think they're looking at today's events when they're not.

---

## Critical

### C1 — "Today" tab stays visually active while showing this week's events
**Where:** `effectiveDateRange` logic + `showingTodayFallback` banner  
**What:** When today has no events, the app silently switches to week view and shows a small amber banner "Nothing today — showing this week instead." But the filter bar still highlights "Today" as the active tab. A user glancing at the bar thinks they're looking at today's events. They're not.  
**Fix:** Either mark "This Week" as active when the fallback fires, or add a visible count badge `Today (0)` so the tab itself communicates the empty state. Don't let the active indicator lie.

### C2 — "View Details" button fires `handleEventClick` twice
**Where:** `App.tsx:2358` (article onClick) + `App.tsx:2425` (button onClick inside same article)  
**What:** The article has `onClick={() => handleEventClick(event)}`. The "View details" button inside the article also has `onClick={() => handleEventClick(event)}`. Clicking the button fires both — the button's handler, then the bubbled click on the article. The panel opens once (idempotent), but `trackEventClick` GA4 fires twice per button click. Your analytics double-count every event detail view that comes from the button.  
**Fix:** Add `e.stopPropagation()` to the button's onClick, or remove onClick from the article and let only the button handle it.

## High

### H1 — No skip-to-content link
**Where:** Page `<head>` / top of render  
**What:** The sticky filter bar has 10+ focusable elements (3 date tabs + 7 category buttons + source dropdown). A keyboard user must tab through all of them on every page load before reaching the first event card. WCAG 2.1 SC 2.4.1 requires a skip link.  
**Fix:** Add `<a href="#main-content" className="sr-only focus:not-sr-only ...">Skip to events</a>` at the very top, and `id="main-content"` on the `<main>` element.

### H2 — All-day events show "—" instead of "All day"
**Where:** Card metadata row, `App.tsx:2402`  
**What:** `event.time || '—'` renders a dash when time is absent. An all-day seminar shows "— · Wheeler Hall" which looks like missing data, not an intentional all-day event.  
**Fix:** `event.time || 'All day'`

### H3 — Away-game events silently filtered out with no explanation
**Where:** `isHomeGame()` applied in `searchOutput` pool filter  
**What:** Sports events at away venues are filtered out in all views. If a student searches for "Cal vs Arizona State" (an away game), they get zero results with the generic "No results found" empty state. There's no indication that away games are excluded, or that there might be more sports events if you look elsewhere.  
**Fix:** Either show away games with an "Away" badge, or add a note in the sports empty state: "Only home games are shown. Visit calbears.com for away schedule."

### H4 — Article cards have no accessible name for screen readers
**Where:** `App.tsx:2350` — `<article>` elements  
**What:** Cards use `<article>` but have no `aria-label` or `aria-labelledby` linking to the event title `<h3>`. Screen readers announce "article" with no context about which event it is.  
**Fix:** Add `aria-labelledby` pointing to each card's title id, or `aria-label={event.title}`.

### H5 — Interpreted-as chips dismiss button is 12×12px (too small)
**Where:** `App.tsx:2274-2284` — chip dismiss `<button>`  
**What:** The × button on "Interpreted as" chips is `h-3 w-3` (12px). WCAG recommends minimum 44×44px touch target. On mobile this chip row appears and the dismiss is practically untappable.  
**Fix:** Increase the hit area — keep the icon at 12px but wrap it in a `p-2` container so the actual tap target is 32px+.

### H6 — Desktop filter bar silently overflows at laptop widths
**Where:** `DesktopFiltersBar` — 3 date tabs + 7 category pills + SOURCE label + dropdown  
**What:** At 1024–1100px viewport width, all these controls likely exceed the container width. The bar uses `overflow-x-auto no-scrollbar` which hides the scrollbar entirely — so users don't know they can scroll horizontally to see more filters. "Entrepreneurship" and "Student Life" pills almost certainly get cut off silently.  
**Fix:** Either reduce the number of visible category pills (add a "More" overflow button), or make the scrollbar faintly visible on desktop.

## Medium

### M1 — Share link is completely undiscoverable
**Where:** `App.tsx` — URL state via `buildUrlStateSearch`  
**What:** The app writes a shareable URL (with query, filters, and selected event ID) into the browser history on every filter change. This is great infrastructure, but there's zero UI to surface it. Users don't know links are shareable and won't think to copy the URL. The detail panel has a "View Official Page" button — a "Copy link" button next to it would leverage the existing URL state with no new engineering.  
**Fix:** Add a copy-link icon button (clipboard icon) in the event detail panel header that calls `navigator.clipboard.writeText(window.location.href)`.

### M2 — Slide-out panel is 450px fixed width — too wide on small screens
**Where:** `SlideOutPanel` — `w-[450px]` on `md:block` (activates at 768px)  
**What:** At exactly 768px viewport width, the 450px panel occupies 59% of the screen. The backdrop is blurred so content is unreadable behind it. A tablet in portrait mode (768px) sees a panel that's nearly the full width.  
**Fix:** Cap at `max-w-[90vw]` or use `w-full md:w-[420px] lg:w-[450px]` to scale down on smaller breakpoints.

### M3 — No "Load more" progress context
**Where:** `App.tsx:2441-2451`  
**What:** The "Load more events (X remaining)" button is clear, but after loading page 2, the user has no idea where they are in the list. If they've loaded 144 of 200 events, there's nothing indicating "page 2 of 3" or a progress bar.  
**Fix:** Add a subtle progress indicator above "Load more": `Showing 144 of 200 events`.

### M4 — Source badge link tap target is 10×10px
**Where:** `SourceBadge` — `text-[10px]` with `w-2.5 h-2.5` icon  
**What:** The source attribution link in the card footer is decorative in size but functional. It links to the source calendar. At 10px text + 10px icon, it's far too small to reliably click, and may be perceived as pure decoration.  
**Fix:** Make the source badge a proper `px-2 py-1` chip so it has enough click area, or remove the link and only show it in the detail panel (where it already appears with better sizing).

### M5 — "Upcoming" label doesn't communicate timeframe
**Where:** Filter bar `DateRanges` array  
**What:** "Upcoming" means "all future events from today." A first-time user doesn't know if that's 2 weeks or 6 months. The event count (`(1,130)`) next to the heading helps, but the label itself is ambiguous — "Upcoming" could mean anything.  
**Fix:** Consider "All Events" or "All Upcoming" to be less ambiguous. Or add a tooltip on hover: "All future events on the calendar."

### M6 — `getGreeting()` is defined but never called
**Where:** `App.tsx:370-375`  
**What:** Dead code. There was presumably a plan to show a time-of-day greeting ("Good morning!") somewhere — likely in the mobile header or hero. It's not used anywhere.  
**Fix:** Either implement it in the mobile header (replacing the static "Cal Events" or adding a line below), or delete the function.

### M7 — Event description doesn't handle newlines
**Where:** `BottomSheet` and `SlideOutPanel` — `<p>{event.description}</p>`  
**What:** LiveWhale event descriptions often contain paragraph breaks (e.g., "Speaker: Dr. Smith\n\nThis talk explores..."). Rendering as a plain `<p>` collapses all whitespace. The description appears as a single block regardless of intended formatting.  
**Fix:** Replace `<p>` with `<p className="whitespace-pre-wrap">` to preserve line breaks from source data.

## Low

### L1 — Error state has inconsistent styling
**Where:** `App.tsx:2248-2254` — `LoadingState.ERROR` block  
**What:** The error card is `bg-red-50` with `text-red-800/600`, but the Retry button is `bg-berkeley-blue`. A Berkeley blue button inside a red error card looks like it belongs to a different component. The error message also says "We couldn't load the latest Berkeley event feed" with no actionable hint (check your connection, try again in a moment).  
**Fix:** Make the Retry button `bg-red-700 hover:bg-red-800` to match the error tone, and add a subtitle: "Check your connection and try again."

### L2 — No end-of-list signal
**Where:** Below the last event card  
**What:** After the last event card (or after clicking "Load more" exhausts all events), the content ends abruptly and jumps straight to the footer. There's no "You've seen all X events" message.  
**Fix:** When `hiddenEventCount === 0 && filteredEvents.length > 0`, show: `All ${filteredEvents.length} events loaded.`

### L3 — Footer "I'm Akhil" copy reduces perceived trust
**Where:** `App.tsx:2463`  
**What:** "I'm Akhil, MBA student at Haas School of Business, and I built this to help Berkeley students..." is genuine and friendly, but for a new user it signals "personal side project" rather than a reliable tool they'll return to. This matters for trust and return visits.  
**Fix:** Lead with the value: "Built for Berkeley students to discover campus events across all schools. By Akhil Neelam, Haas MBA." The information is the same; the framing puts the product first.

### L4 — Status banner is dismissible by nobody
**Where:** `App.tsx:2219-2229`  
**What:** The yellow "Showing mostly fresh data" warning banner has no dismiss button. It persists for the entire session even after the user has acknowledged it. Persistent warnings lose their urgency.  
**Fix:** Add an × dismiss button. Store the dismissed-until timestamp in sessionStorage so it doesn't reappear on the same visit.

### L5 — No back-to-top after loading more events
**Where:** After clicking "Load more" with 200+ events  
**What:** Loading more appends events below. There's no "Back to top" affordance. Users who load 3 batches and want to re-filter must scroll all the way back up.  
**Fix:** Show a floating "↑ Top" button when the user has scrolled past 800px (appears after the first batch of cards).

---

## Positive Findings

Things that work well and shouldn't change:

- **Empty states are exceptional.** Every empty condition has a contextual title, an explanation, and 1–2 specific action buttons. The logic handles 7+ distinct empty state scenarios. This is best-in-class.
- **Search interpretation chips are a great UX pattern.** Showing "Interpreted as: This Week, Science & Tech" with per-chip dismiss gives users control without requiring them to learn query syntax.
- **Today-fallback-to-week is the right behavior** — the problem is only the filter UI not reflecting it (C1 above).
- **The detail panel close-on-backdrop-click is well implemented.** The `useDialogAccessibility` hook handles focus trapping, scroll lock, Escape key, and focus restoration correctly.
- **URL state persistence is comprehensive.** Search query, all filters, and selected event ID all round-trip through the URL. This is more than most event apps do.
- **The `runAdapterWithTimeout` pattern** means a single slow source never blocks the full page load.
- **Card stagger animation** (`animate-card-in` with delay up to 500ms) feels premium without being excessive. The `prefersReducedMotion` check is correctly implemented.
